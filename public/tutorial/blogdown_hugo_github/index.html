Here is the workflow:

Go to Github

  • Create Username.github.io repo in Github.

  • Clone it to your local computer (just one folder down in the hierarchy UserWebsite.)
    • If this is your first time using Github in your machine, authenticate to get access to your Github via url.
$cd /path/to/one/foler/above/the/clone (in my case path to UserWebsite)

$git clone your/url/of/the/repo #still use url
  • remove README.md in the clone Username.github.io.

Go to Rstudio

  • Open R Studio.

  • Create a new project and choose ‘existing directory’ (UserWebsite) that is above the working directory (Username.github.io) that you created (clone) from Github.

    • Keep working directory Username.github.io completely empty to create a site.

  • Install blogdown from Github:

devtools::install_github("rstudio/blogdown")
library(blogdown)
  • If you need a latest version of hugo
    • If you install hugo the first time, install Homebrew.
# First check whether you already have hugo
hugo_version()
install_hugo(force = TRUE)
  • In R Studio set the working directory to your clone Github directory.
setwd("Username.github.io")
  • Choose a theme.
blogdown::new_site(theme = "gcushen/hugo-academic", theme_example = TRUE)
blogdown::hugo_version()
  • I had install Hugo independently of install_hugo so I received the following warning. Then, I deleted one of them.

For example, the warning mesagge was

Warning message: In find_exec(“hugo”, “Hugo”, “You can install it via blogdown::install_hugo()”) : Found hugo at “~/Library/Application Support/Hugo/hugo” and “/usr/local/bin/hugo”. The former will be used. If you don’t need both copies, you may delete/uninstall one.

unlink('~/Library/Application Support/Hugo', recursive = TRUE)

Deploying at Github

Note: The following steps are from Amber Thomas to keep the flow in this article.

  1. Go to your repo on GitHub. Go to Username.github.io.

On the left-hand side you’ll see a button that reads “Branch: Master”. Click on it and in the text box type “sources” and click “Create branch: sources from ‘master’”.

  1. Now you have 2 branches: ‘sources’ and ‘master’. You should see a button next to “1 commit” that reads “2 branches”. Click on it.

  2. You should see that ‘master’ is your default branch and ‘sources’ is your active branch. We want to switch these, so click the button that says “Change default branch”.

  3. Under Default Branch, choose ‘sources’ from the dropdown menu and click update. It will ask if you are sure. If you have followed each step so far and have nothing on your master branch, then click “I understand, update the default branch”.

  4. Your default branch is now ‘sources’. In your command line interface (terminal or git bash) navigate to your github working directory (for me, it’s Username.github.io), and type the following to download your changes to your local machine.

git fetch

then type,

git checkout sources

Now we need to set up a Subbranch in the Github repository so that you will push all the files to sources and the public folder to master.

  1. Set up a Subbranch.

copy and save setup.sh in your Username.github.io directory.

(copy setup.sh and change your $USERNAME and $SOURCES)

make sure nothing in your master branch because setup.sh will delete everything in the master branch.

bash setup.sh (only run one time)

If you get to authenticate your access to Github repo via ssh (earlier you might have used url)

  • authentication steps

  • ssh key to github and known_hosts: adding your ssh key to the ssh-agent

  • Go to terminal and the directory should be `Username.github.io’

ssh -T git@github.com
  • then say ‘yes’

After Setup

  • add posts or modify local files (or don’t do anything)
  • open config.toml and set : baseurl = “https://Username.github.io/”

Go to terminal and push changes to Github remote

git status 
git add .
git commit -m "default page"
git push

You have pushed all the changes in the local computer to sources.

Copy and save deploy.sh file into your local computer Username.github.io directory.

bash deploy.sh

  1. This will pull the public folder in master branch to sources branch (because we have just now push local changes to sources branch.)

  2. Building the website (inside the Username.github.io) by calling: hugo

  3. Then, push updated public folder to sources branch.

  4. Then, pushing updated public folder to master branch.

Adding changes

  • modify post or add posts or modify pages.
git status
git add.
git commit -m "changes made"
git push
bash deploy.sh

Note:

sources branch will contain all the contents in the Username.github.io and the public folder in the sources will be pused to master branch.

Reopen R studio to update posts

  • Click your UserWebsite.Rproj
  • set the current directory to your working directory:
setwd("Username.github.io")
  • load blogdown
library(blogdown)
  • go to Addins choose Serve Site or call
serve_site()

  • After you make changes to your content folder, LiveLoad will automatically serve your site.

  • go to terminal

git add .
git commit -m "another edit"
git push
bash deploy.sh
  • now changes will be made to your website.

Adding your Github pages for your R package (minimal work)

  • After creating your R pacakage, install pkgdown
devtools::install_github("r-lib/pkgdown")
library(pkgdown)
pkgdown::build_site()

  • create _pkgdown.yaml' indocs/` folder

  • direct terminal to docs/ folder

touch _pkgdown.yaml
  • add the following to _pkgdown.yaml

url: http://Username.github.io/pkgname

  • save

  • push the commits (docs/ forlder to master branch)
    • go to terminal
git add .
git commit -m "docs/ folder for Github package"
git push origin master
  • Go to Github account
    • Go to the repository
    • Go to settings
    • Scroll down to Github pages section:
      • Under the source menu choose master branch/docs folder
      • Save
  • That’s wow!
  • You can give the url https://Username.github.io/pkgname/ to your site created using blogdown.
